Сжатие данных. Алгоритм Хаффмана

В ситуациях, когда нам нужно уменьшить объём памяти, занимаемый данными, мы прибегаем к сжатию данных. Это особенно полезно при хранении данных и их пересылке.

При сжатии данных мы можем как потерять часть исходной информации, так и сохранить её в исходном виде. Например, в формате изображений JPG используется сжатие с потерями, а в формате PNG — без потерь.

В этом уроке мы рассмотрим один из самых простых алгоритмов сжатия без потерь — алгоритм Хаффмана.

Основная идея алгоритма — кодирование часто встречающихся данных наименьшим количеством бит, и, наоборот, кодирование редко встречающихся данных более длинной последовательностью бит. При этом код должен быть префиксным — ни один полученный код не должен быть префиксом другого кода, то есть ни один код не может быть началом другого кода. Например, код 001 является префиксом для кода 0010. В данном алгоритме такое не допускается.

Рассмотрим на примере кодирования строки foo bar baz. Если сохранить её в формате UTF-8, каждый символ будет занимать 8 бит. Всего у нас 11 символов, поэтому, чтобы сохранить эту строку, нам потребуется 8 * 11 = 88 бит. Теперь попробуем её сжать.

Прежде всего нам нужно определить частоту повторения каждого символа в строке:

Символ Частота повторений
‘f’ 1
‘o’ 2
’ ‘ 2
‘b’ 2
‘a’ 2
‘r’ 1
‘z’ 1

Теперь присвоим каждому символу свой код. Символы, которые встречаются чаще, получат наименьший код. Для это построим бинарное дерево Хаффмана, следуя алгоритму:

  1. Для каждого символа создаём ноду будущего дерева, значение которой равно этому символу. Помещаем их в очередь с приоритетом, где наиболее часто повторяющиеся символы имеют больший приоритет:

Очередь с приоритетом для символов из исходного сообщения

  1. Берём два наименее приоритетных элемента из очереди и объединяем их в дерево. Для этого создаём пустую ноду, а два элемента из очереди делаем её детьми:

Новый узел дерева из двух символов с самыми низкими приоритетами

  1. У созданного дерева частота повторений равна сумме повторений двух составных элементов. Помещаем это дерево обратно в очередь соответственно весу:

Помещаем новый узел на место в очереди

  1. Продолжаем с пункта 2 до тех пор, пока в очереди не останется один элемент. Это и будет дерево Хаффмана:

Продолжаем объединение частей дерева в одно целое дерево Хаффмана

Затем обойдём дерево, добавляя 1, если идём влево, и 0, если вправо:

Дерево Хаффмана

В листьях дерева лежат символы, а путь до листа содержит его код:

Символ Код
‘f’ 100
‘o’ 111
’ ‘ 110
‘b’ 01
‘a’ 00
‘r’ 1011
‘z’ 1010

Символы, которые повторяются, чаще имеют меньшую последовательность бит. Перепишем исходное сообщение, используя эти коды: 1001111111100100101111001001010. Сжатая строка занимает 31 бит против 88 исходных.

Чтобы расшифровать код, мы также пойдём по дереву, сворачивая в нужную сторону в соответствии с текущим битом: 0 или 1. Когда дойдём до листа, начнём всё с начала. Например, пройдя по первым трём битам, найдём символ f:

Поиск символа по пути в дереве Хаффмана

Также это дерево можно представить хеш-таблицей, где битовые последовательности — ключи, а символы — значения. Тогда, продвигаясь по закодированному сообщению бит за битом, мы будем искать символ в таблице по последовательности бит до тех пор, пока не получим соответствие. Затем начнём всё с начала в поисках следующей последовательности:

Поиск по хеш-таблице с кодами для символов

Чтобы расшифровать закодированное сообщение, нужно отправить вместе с ним таблицу частот для построения дерева либо готовую карту с кодами каждого символа. Это может свести на нет выигрыш от сжатия информации.

Например, в нашем коротком сообщении длиной 31 бит дополнительно придётся переслать по 8 бит для каждого из семи символов, что уже даст 31 + 7 * 8 = 87, то есть всего на 1 бит меньше исходного сообщения. А это мы ещё не посчитали частоты или готовые коды для каждого символа. В последнем случае нам понадобится ещё 21 бит (сумма кодов для каждого символа), что даст суммарно на 20 бит больше исходного сообщения.

Это основной недостаток такого метода шифрования. Поэтому алгоритм Хаффмана лучше применять только для больших сообщений.

Ещё один недостаток в том, что нужно дважды проходить по сообщению — один раз для сбора частот, второй раз для кодирования.

С другой стороны, этот алгоритм просто понять. Он считается базой для более сложных его вариаций, которые исправляют некоторые его недостатки. Также знание алгоритма Хаффмана может пригодиться на собеседовании. 🙂
