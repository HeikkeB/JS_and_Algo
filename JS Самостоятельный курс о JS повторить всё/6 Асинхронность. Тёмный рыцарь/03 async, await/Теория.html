async, await

Вернёмся к примеру с промисами и последовательными запросами. Такой код выглядит намного лучше, чем “callback hell”:

http.post('/api/v1/signin', user)
	.then(resp1 => http.get(`/api/v1/chats/${resp1.id}`))
	.then(resp2 => http.post(`/api/v1/chats/${resp1.id}/messages`, {info: resp2}))
	.then(...)
	.catch(error => console.error(error));

Но ведь работать с синхронным кодом гораздо понятнее и удобнее:

try {
	const resp1 = await http.post('/api/v1/signin', user);
	...
}
catch (error) {
	console.error(error);
}

Ключевое слово await указывает на то, что нужно дождаться выполнение промиса. Если он завершится успешно, в переменную будет записан результат resolve, иначе — возникнет исключение, которое хорошо бы поймать.

Нельзя использовать await без обозначения функции async. Они всегда работают в паре:

const userSigning = async () => {
	try {
		const resp1 = await http.post('/api/v1/signin', user);
		...
	}
	catch (error) {
		console.error(error);
	}
};

// или

async function userSigning() {
	try {
		const resp1 = await http.post('/api/v1/signin', user);
		...
	}
	catch (error) {
		console.error(error);
	}
}

Функция, помеченная как async, — всегда возвращает промис.

async function simpleAsync() {
	console.log('hello');
	return 42;
}

simpleAsync() // hello
	.then(data => { console.log(data) }); // 42

await — это ключевое слово, которое умеет резолвить не только промисы:

async function simpleAsync() {
	console.log('hello');
	const data = await 42;
	return data;
}

simpleAsync() // hello
	.then(data => { console.log(data) }); // 42

Вернуть await

Делать return await — плохая затея. Функция с async всегда возвращает зарезолвенный промис или выкидывает ошибку. Поэтому await заберёт дополнительное время, перед тем как сам промис разрезолвится или упадёт.
