Очередь событий

Асинхронность, параллельное выполнение запросов и setTimeout работают благодаря очереди запросов. Этот урок поможет понять механизм очереди.

Разберём работу этого кода:

console.log('first'); // 1
// 2
setTimeout(function callback() { // 3
	console.log('third'); // 4
}, 3000); // 5
// 6
console.log('second'); // 7

Понимать очередь запросов — значит в каждый момент времени знать, какие запросы находятся в стеке, какие — в очереди, а какие уже отработали и оставили след в консоли:

Шаг 1

Начнём разбирать код с самой первой строчки.

В начале на стек попадёт IIFE как вызывающая функция, а потом console.log('first'):

Вывод в консоль не запускает внутри себя ещё одну функцию. Ведь мы рассматриваем не внутренние вызовы в языке, а функции, определённые разработчиком.

Поэтому стек сразу начнёт раскручиваться. То есть в консоли появится строка 'first', а из стека удалится console.log, который вернёт управление IIFE. Указатель приведёт программу ко второй строке:

Шаг 2

Интерпретатор встречает на своём пути setTimeout. Он знает, что setTimeout — это асинхронная функция, которая выполнится через заданный промежуток времени. В данном случае — через три секунды.

Однако setTimeout такая же функция, как и все остальные, поэтому она попадёт на стек:

В момент исполнения setTimeout будет определённо следующее:

  1. Через три секунды должно быть выполнение функции callback.
  2. Выполнение асинхронное и неблокирующее.

Поскольку setTimeout не блокирует выполнение работы, он добавляет исполнение callback в специальный «планировщик» (от англ. scheduler). Для этого используется WEB API. Через три секунды планировщик переместит callback в очередь событий.

Таким образом, после выполнения setTimeout произойдут следующие события:

  1. setTimeout добавит в планировщик callback.
  2. Вернёт управление IIFE, но не будет выполнять callback.

Шаг 3

Интерпретатор придёт к строке console.log('second'). Всё отработает как в шаге один.

Вывод строки займёт меньше трёх секунд — console.log простая, нативная и синхронная операция. Поэтому произойдут ровно такие же два шага, как в начале, но с одним исключением. Теперь своей очереди будет ждать функция callback:

После вывода в консоль строки 'second', console.log вернёт управление главной функции IIFE:

Интерпретатор увидит, что стек ещё не пуст и продолжит исполнять IIFE. Но после седьмой строчки ничего нет, поэтому IIFE вернёт подобие null. Если говорить точнее, то вернётся указатель на успешный выход из программы. И стек останется пуст:

Очередь событий ещё не пуста. Поэтому интерпретатор и планировщик продолжат выполнять (или ожидать) код, пока не произойдут все события из очереди.

Шаги 1–3 проходили примерно на первой секунде. Но через три секунды должно произойти выполнение функции callback. То есть Web API планировщика, в котором callback ожидает завершение этого времени, будет перемещён в очередь событий.

Шаг 4

Спустя три секунды функция callback попадёт в очередь событий:

Интерпретатор поймёт, что очередь не пуста, и выгрузит из неё на стек функцию callback.

После попадания на стек, всё начинается как обычно — для функции запускается стек и последовательно выполняются команды на аналогичном стеке, но уже для неё.

Наконец выполнится console.log('third'):

На последнем шаге из стека выталкивается callback, который возвращает управление программе, и последняя завершает выполнение.

Сервис Loupe наглядно показывает перемещение вызовов между стеком и очередью. Там уже опубликован код этого урока. Одна особенность: Loupe игнорирует IIFE, если она не вызвана явно. Для лучшего понимания процесса оберните код в самовызывающуюся функцию:

(function () {
  console.log('first');

	setTimeout(function callback() {
		console.log('third');
	}, 3000);

	console.log('second');
})()
